硬件平台:
Ubuntu 16.04
Intellij IDEA Ultimate

          maven翻译为“专家”，“内行”。它是Apache公司下的一个纯Java开发的开源项目，其本质上是一个项目管理工具，我们可以使用maven对Java项目进行构建、依赖管理，并且现在越来越多的开发人员使用maven进行开发项目。

项目管理工具 - Maven

maven是基于项目对象模型（POM）的工具，可以通过一小段描述信息来管理项目的构建、报告和文档的项目管理工具，其有两大核心功能：
1. 依赖管理
Java中的依赖就是Java项目的运行需要来自第三方的jar包的支持才可以，这样就称为该Java项目依赖于这些第三方的jar包。
maven的依赖管理功能就是对项目所有依赖的jar包进行管理的过程。
传统的项目工程要管理所依赖的jar包完全考开发人员进行操作，手动的获取jar包并添加到项目中去，这样可能会出现以下问题：

  1. 没有对jar包版本的统一管理，容易导致版本冲突；
  2. 从网络上查找jar包非常不方便，甚至有些可能都找不到，
  3. 将jar包直接放在项目目录中会导致工程很大；

maven项目管理所依赖的jar包不需要手动向工程中进行添加jar包，而只需要在配置文件pom.xml中添加jar包的坐标，maven便可以自动从仓库中下载对应的jar包工项目使用：

maven 依赖管理的好处：

  1. 通过pom.xml配置文件可以对jar包的版本进行统一管理，避免版本冲突；
  2. maven 的中心仓库中的jar包十分齐全，并可以自动下载，非常方便；

  1. 一个命令便可完成项目构建、运行，方便快捷；
  2. maven对每个构建阶段进行规范，有利于大型团队协作开发；

Maven 的三大仓库

maven 中有三个仓库，分别是：

  1. 本地仓库
    用来存储从远程仓库或中央仓库中下载的插件或是jar包，项目中需要的插件和jar包，会优先从本地仓库查找。
  2. 远程仓库（私服）
    如果项目需要的插件或jar包，本地仓库没有，默认去远程仓库进行下载，远程仓库可以在互联网也可以在局域网内，远程仓库是属于一个团体的这样的概念，比如一个团队或是一个公司常用的插件和jar包组成一个远程仓库。
  3. 中央仓库
    这是maven团队维护的一个面向互联网的最大型的仓库，包含了世界上大部分的流行的开元项目构建。

Maven 的目录结构和命令

目录结构：
使用maven创建的工程就称为maven工程，maven工程具有一定的目录结构规范，如下：
src/main/java：放置 .java 文件
src/main/resources：放置项目资源文件，如：各种配置文件
src/main/webapp：页面资源文件，如：js、css、图片等
src/test/java：放置所有单元测试的 .java文件，如：JUnit测试类
src/test/resources：放置测试资源文件
target：项目输出位置，.jar、.war和所有编译后的.class文件都输出到此目录
pom.xml：maven项目的核心配置文件

maven 常用命令：
每个构建项目的命令都对应了maven底层的一个插件。
compile： 是maven工程的编译命令，作用是将src/main/java目录下的文件编译为.class文件输出到target目录下；

test： 是maven工程的测试命令，会执行src/test/java目录下的所有单元测试类；

clean： 是maven工程的清理命令，执行clean会删除target目录下的内容；

package： 是maven工程的打包命令，对于Java工程执行package打包成jar包，对于web工程打包成war包；

install： 是maven工程的安装命令，执行install将maven工程打包成jar或war并发布到本地仓库中；

Maven 的三套生命周期

maven 对项目构建过程分为三套相互独立的生命周期，分别是：
Clean Lifecycle：在进行真正的构建之前进行一些清理工作
Default Lifecycle：构建的核心部分，包括：编译、测试、打包、安装、发布
Site Lifecycle：生成项目报告，站点，发布站点；
每个maven命令都对应这生命周期的某个阶段，执行命令会将该命令所在的生命周期之前的阶段自动执行，并且执行某个生命周期的某个阶段是不会影响其他的生命周期。

使用maven的好处：

  1. 一步构建
    maven 对项目构建的过程进行标准化，通过一个命令便可以完成工程构建；
  2. 依赖管理
    maven 工程不用开发者手动导入jar包，通过在pom.xml配置文件中添加坐标便可从maven中心仓库中自动下载对应的jar包，方便且不易出错；
  3. maven 跨平台
    maven可以实现跨平台，主要服务于基于Java平台的项目构建、依赖管理和项目信息管理，可在Windows、Linux等操作系统上使用；
  4. 开发效率高
    maven 遵循规范开发有利于提高大型团队的开发效率，降低项目的维护成本；

依赖范围

有两个工程A和B，如果A依赖与B，则需要在A的pom.xml文件中共添加B的坐标，添加坐标的时候需要指定依赖范围，依赖范围包括：

  • compile：编译范围，指A在编译时依赖B，此范围为默认的依赖范围。编译范围的依赖会用在编译、测试、运行等过程中，由于运行时需要，所以编译范围的依赖包会被打包。
  • provided：只有在JDK或者容器中已经提供了该以来之后才可以使用，provided依赖在编译和测试时需要，在运行时不需要，。
  • runtime：在运行和测试系统的时候需要，但在编译的时候不需要，由于运行时需要所以会被打包。
  • test：在编译和运行时都不需要，他们只有在测试编译和测试运行阶段可用，由于运行时不需要所以不会被打包。
  • system：和provided类似，但是必须显示的提供一个对于本地系统中jar文件的路径，需要指定systemPath磁盘路径，system依赖不推荐使用。

依赖范围测试总结：
默认引入的jar包——compile【默认范围，可以不显式写出，编译、测试、运行都有效】
servet-api、jsp-api ——- provided【编译、测试有效，运行时无效，避免和tomcat下的jar包冲突】
JDBC驱动jar包 ——– runtime【测试、运行有效】
JUnit ——- test【测试有效】
依赖范围由强到弱的顺序是：compile > provided > runtime > test

聚合

聚合一般是将一个工程拆分成多个模块开发，每个模块又是一个独立的工程，但在运行时必须将所有模块聚合到一起才是一个完整的工程，此功能可以使用maven的聚合工程实现。
例如电商项目，包括商品模块、订单模块和用户模块等，就可以对不同的模块单独创建工程，最终在打包的时候将不同的模块聚合到一起就可以了。
